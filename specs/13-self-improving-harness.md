# 13 - Wave 4: the harness improves itself

**Goal:** the loop gains the machinery to measure, regression-test, and tune ITSELF: pinned definitions, trajectory replay, speculative attempts, risk-tiered review, seeded-defect canaries, model-drift detection, and distilled playbooks. Program Wave 4.

## Design

**Unit 1 - definition pinning.** `RunStarted` gains a definition hash (workflow.yml + the full agent-prompt set, canonicalized). A `rigger step` whose on-disk definition hashes differently HALTS loudly naming the drift, with an explicit `--rebase-definition` escape that records the supersession (old hash, new hash) before continuing. New runs are always free (R1's edit-to-reconfigure holds for run boundaries; only LIVE runs pin). Owns: definition integrity. This closes the sharpest exposure the survey found: mid-campaign prompt edits silently changing replay semantics.

**Unit 2 - trajectory replay/eval, after unit 1.** `rigger replay <run-id> --against <config-rev>`: re-drive a completed run's spec through a candidate config (workflow + prompts at a named git rev) in an isolated scratch namespace, then print the stats diff against the original run's recorded metrics. Past runs become the regression corpus for config edits - "did that prompt change regress first-pass yield?" gets an answer without a live campaign. Deterministic seams (run scoping, pinned definitions) make the baseline citable. Owns: config-edit evaluation. Exclusion: it never writes into the real project namespace.

**Unit 3 - first-green-wins speculation.** Unit classes may declare `speculation_width: K`: the conductor parks K parallel implementer attempts (distinct deterministic spawn ids in one speculation group) in isolated worktrees; the first candidate to pass gates AND adjudication wins, the rest are cancelled - all recorded (group id, winner, losers) as metadata on existing events. Width and budget interaction are config; the spawn breaker remains the global cap and allocation is logged. Owns: intra-unit parallel attempts. Exclusion: model laddering (spec 10) composes but is not re-implemented; review depth is unit 4's.

**Unit 4 - risk-tiered review depth.** A declarative `review_tiers:` map in workflow.yml routes units to review depth by observable risk (blast-radius size, first-pass gate outcome): low-risk units run lenses + adjudicator; high-risk or gate-flapping units add the adversary. The ADJUDICATOR and full gates are mandatory on every tier - only the adversary flexes. Every tier decision is logged with its inputs. Shipped default: everything gets the full panel (opt-in tiering), so behavior is unchanged until a workflow declares tiers. Owns: review-depth routing. Exclusion: no learned/bandit tuning in this unit - the shipped mechanism is the static declarative map; adaptive tuning waits for evidence from spec 11's telemetry.

**Unit 5 - seeded-defect canary corpus, after unit 4.** A versioned corpus under `canaries/` (known-good micro-units + cataloged planted defects drawn from the adversary's hunt list) and a `rigger canary` command that runs the review panel against each, scoring: which tier caught the defect, whether the adjudicator's verdict was correct, and verdict stability under finding-order shuffling (position-bias probe). Results land in the event log under a canary namespace; `rigger stats --canary` reports catch rate by tier. This is the loop's only RECALL measurement - everything else measures precision. Owns: judge-the-judges ground truth. Exclusion: automatic scheduling on model change is unit 6's trigger.

**Unit 6 - model-change drift monitor, after unit 5.** When a tier's resolved model id (Gap-6 stamps) differs from the previous run's, `rigger validate` warns and recommends a canary run, and `rigger canary --if-model-changed` (unit 5's canary, gated on that drift) runs only then - the drift monitor for silent alias re-points. Owns: model-drift DETECTION and its canary trigger. Exclusion: the canary corpus and scoring are unit 5's; cross-run lesson distillation is unit 7's - this unit adds no playbooks and touches no prompt injection.

**Unit 7 - distilled playbooks, after unit 5.** A post-run distiller folds the run's `LessonLearned` events into a deduplicated playbook pool (markdown + frontmatter with trigger predicates, rigger's native agent-file shape) stored as derived artifacts under `.rigger/playbooks/` and injected by blast-radius relevance INSIDE the existing per-section prompt budgets (replacing pure recency for the lessons section). The pool is a rebuildable projection of the log - never hand-edited state; `rigger playbooks --rebuild` reconstructs it. Owns: cross-run LESSON distillation and the lessons-section injection ranking. Exclusion: decisions/findings injection (spec 07's budgeted sections) keeps its semantics - only the lessons slice gains relevance ranking; model-drift detection and its canary trigger are unit 6's.

## Global constraints

- Hyphens, not em dashes. New event types: NONE - hashes, groups, tiers, canary scores, and distillation ride as metadata on the existing vocabulary; playbooks are derived files, not events. Shipped defaults change NO current behavior (pinning warns loudly only on live-run drift; tiering defaults to the full panel; speculation defaults off). Both lanes green; replay determinism holds for every new spawn id (speculation group members derive deterministically).

## Done when

- [ ] a run pins its definition hash at start, a live-run step under a drifted definition halts loudly, and `--rebase-definition` records the supersession and continues - pinned including the new-run-is-free path
- [ ] `rigger replay <run-id> --against <rev>` re-drives a completed run's spec under the candidate config in an isolated namespace and prints the stats diff against the recorded baseline, never touching the real project streams - pinned with a fixture run
- [ ] a unit class with `speculation_width: K` parks K deterministic parallel attempts, the first gate-green adjudicator-approved candidate integrates, the rest are cancelled, and the group/winner/losers are logged - pinned with K=2, including budget accounting
- [ ] a declared `review_tiers:` map routes units to lenses-only or full-panel by blast-radius and gate outcome with the adjudicator and gates mandatory everywhere, every routing decision logged, and the shipped default unchanged-full-panel - pinned for both routes
- [ ] a canaries corpus and `rigger canary` score per-tier catch rate, adjudicator correctness, and order-shuffle stability into a canary namespace reported by `rigger stats --canary` - pinned with at least three cataloged defect classes
- [ ] a resolved-model change (a tier's stamped model id differing from the previous run's) triggers a `rigger validate` warning and runs `rigger canary --if-model-changed`, while an unchanged model runs no canary - pinned with a seeded model-id change AND a no-change control
- [ ] the post-run distiller folds `LessonLearned` events into a deduplicated, trigger-scoped playbook pool under `.rigger/playbooks/` injected by blast-radius relevance within the existing lessons budget (replacing pure recency for that section), and `rigger playbooks --rebuild` reconstructs the pool from the log - pinned including the rebuild and the relevance-over-recency ordering
