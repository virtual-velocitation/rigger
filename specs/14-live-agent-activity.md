# 14 - Wave 5: rigger presents live agent activity

**Goal:** close the observability blind spot where an agent works for 20-30 minutes emitting nothing an observer can see. Rigger already holds every signal - run-stream milestones and per-spawn liveness markers - but scatters them: the dash reads only milestones (a blackout between them), and the sandboxed JS driver reconstructs liveness one file at a time by spawning an LLM to `stat` a marker. Rigger becomes the AUTHORITY that consolidates its signals and PRESENTS a live per-agent view; every consumer receives it rather than reassembling it. Program Wave 5 (Gap 27).

## Design

**Unit 1 - the progress store and emit path.** A new per-spawn telemetry event `AgentProgress{id, unit, stage, activity, seq}` records what an agent is doing at a significant step (a tool call, a build, a commit, a decision). It lands in a SEPARATE store - `.rigger/progress.db`, a sibling of the existing `.rigger/graph.db`, per-project namespaced through the same identity decorator - never the run stream. A new `rigger progress <id> "<activity>"` CLI verb (a sibling of `rigger emit`/`rigger result`) appends one such event. Because progress lives in its own store, NO run fold (the ledger, the spawn frontier, the metrics, the conductor, the context graph) can read it: the run stream, its projections, and replay are byte-identical whether or not any progress was ever emitted. Owns: the progress store, the event shape, the emit command, and the replay-isolation guarantee. Exclusion: consolidating/reading it is unit 2's; the emit cadence (who calls it, when) is unit 3's; dash rendering is unit 4's.

**Unit 2 - the consolidator and its Rust presentation surfaces, after unit 1.** A pure Rust consolidator folds the CURRENT run's three signals into one live view: run-stream milestones (each in-flight spawn's last lifecycle event), the progress store (its latest activity and how long ago), and the spec-10 liveness marker mtime (how long since its last heartbeat). For each in-flight spawn - a parked spawn with no recorded result, the `spawn::step_result` frontier - it yields `{id, unit, stage, latest_activity, activity_age_s, liveness_age_s, last_milestone}`. Two first-class presentation surfaces expose it: `rigger status [--json]` (a human table by default, machine JSON with `--json`) and a `rigger_activity` MCP tool on `rigger serve` returning the same view. rigger reads the marker mtime in Rust here, so NO consumer ever stats it. Owns: the consolidator fold and BOTH Rust presentation surfaces (the `status` CLI and the `rigger_activity` MCP tool). Exclusion: the DRIVER-side consumption (workers emitting progress; drivers dropping the probe) is unit 3's; the browser rendering is unit 4's; the progress store is unit 1's.

**Unit 3 - driver-side emit and probe retirement, after unit 2.** Two halves, both on the driver boundary. (a) EMIT: every driver's worker-framing prompt - the native `/rigger` workflow (`workflows/rigger.js`), the standalone shim (`.rigger/shim/shim.mjs`), and the CLI driver's worker prompt - instructs the worker to run `rigger progress <id> "<one line of what you just did>"` after each significant step, ADDITIVE to the existing liveness-marker touch. (b) RETIRE: the JS drivers STOP spawning the haiku `stat` probe for liveness and consume the presented `liveness_age_s` (from `rigger_activity` / `rigger status`) instead. The spec-10 marker itself is untouched - the Rust `rigger step` sweep still reads it and still guards hung agents; only the driver-side probe is removed. Owns: the worker emit instruction across all three drivers and the JS probe retirement. Exclusion: the `rigger_activity` tool and the consolidator are unit 2's; the marker, its sweep, and its watchdog (spec 10 unit 3) are unchanged.

**Unit 4 - the dash present-view, after unit 2.** The dash's "present" view renders the consolidator: for each in-flight spawn, its stage, latest activity, activity-age, and liveness-age, so the 26-minute blackout becomes "unit4 implementer - grep #12: conductor.rs compensation path, 20s ago". `build_state` reads the current run's progress store alongside the run stream, and the live page refreshes the panel through the existing `events_json(since)` incremental feed - no reload. Owns: the dash rendering of the live per-agent view. Exclusion: the consolidator fold and its CLI/MCP surfaces are unit 2's; the progress store is unit 1's.

## Global constraints

- Hyphens, not em dashes. The `run` stream vocabulary is UNCHANGED: `AgentProgress` is a new type in a SEPARATE store, so it adds nothing any run fold (ledger/spawn/metrics/conductor/context-graph) reads, and replay determinism is STRUCTURALLY preserved - a separate store cannot enter a run fold. The spec-10 liveness marker, its Rust sweep, and its watchdog are untouched; only the driver-side haiku probe is removed.
- Both CI lanes stay green: `cargo fmt --check`; `cargo clippy --all-targets -- -D warnings`; `cargo test`.
- GATE-BLIND SURFACES (handbook rule 4): the driver changes in `workflows/rigger.js` and `.rigger/shim/shim.mjs` are JavaScript the cargo gates never compile or run. For unit 3 the adjudicator MUST demand explicit evidence - the exact prompt text added and the exact probe code removed - not a green gate, which proves nothing about an untested artifact.

## Done when

- [ ] `rigger progress <id> "<activity>"` appends one `AgentProgress` event to `.rigger/progress.db` and NEVER to the run stream, and a run's ledger/spawn/metrics projections AND its replay are byte-identical whether or not progress events were emitted - pinned by a store-target test (the event lands in progress.db, absent from the run stream) AND a fold-equality test (the same run stream projects identically with progress present vs. absent).
- [ ] the consolidator yields, for each in-flight spawn, its stage + latest activity + activity-age + liveness-age + last milestone, folded from the run stream + progress store + marker mtime - pinned with a seeded fixture: an in-flight spawn carrying progress events and a liveness marker of a known age.
- [ ] `rigger status --json` prints that consolidated view for the current run's in-flight agents, and `rigger serve` answers a `rigger_activity` MCP call with the same view - pinned by asserting the JSON shape/values over the fixture (CLI) and by an MCP request/response test (so the shim obtains live agent state over MCP with no filesystem access).
- [ ] every driver's worker prompt instructs per-significant-step `rigger progress` emission, and the JS drivers no longer spawn the haiku liveness `stat` probe (consuming the presented `liveness_age_s` instead), while the spec-10 marker and its Rust sweep remain - since these artifacts are outside the cargo gates, the adjudicator demands the concrete prompt-text and probe-removal evidence rather than accepting a green gate.
- [ ] the dash present-view renders each in-flight agent's live activity + age from the consolidator and refreshes it through the incremental feed - pinned by a render test over seeded progress asserting the activity string and age appear in the emitted state/HTML.
